# Radiology Rush（《影像科的日常》）游戏设计文档

## 1. 项目概述

### 项目名称
- 中文名：《影像科的日常》
- 英文名：Radiology Rush

### 游戏类型
- 2D 网页休闲模拟经营游戏
- 实时流程调度（类似《胡闹厨房》《沙威玛传奇》）
- 轻度时间管理 + 队列处理 + 资源调度

### 目标平台
- Web（PC 优先）
- 后续兼容移动端 H5

### 技术方向
- 前端：React + Vite + TypeScript
- UI：TailwindCSS + shadcn/ui
- 动画：Framer Motion
- 状态管理：Zustand
- 音频：Howler.js
- 部署：OSS/COS + CDN

---

# 2. 核心创意

玩家扮演医院影像科工作人员。

每天需要：

- 接待病人
- 分配检查室
- 控制排队
- 应对急诊
- 防止设备爆满
- 避免病人情绪崩溃
- 保持科室运转

核心体验：

> “手忙脚乱但又停不下来。”

游戏整体氛围：

- 忙乱
- 轻松搞笑
- 有医院真实感
- 带一点现实吐槽
- Q版风格

---

## 短期目标

- 处理更多病人
- 提高评分

## 长期目标

- 解锁新设备
- 解锁新病人类型

- 解锁夜班模式

# 4. 游戏循环（Core Loop）

## 基础循环

病人进入
→ 玩家点击病人
→ 分配检查室
→ 检查耗时
→ 完成检查
→ 获取分数
→ 新病人继续进入

玩家需要：

- 快速判断病人类型
- 分配正确设备
- 控制排队节奏
- 优先处理急诊
- 避免设备空转

---

# 5. MVP（第一版最小可玩版本）

## MVP目标

做出一个：

> “可玩、会上瘾、能体现影像科忙乱感”的网页小游戏。

第一版不做：

- 登录
- 云存档
- 联机
- 排行榜
- 复杂成长系统
- 大地图

# 6. MVP玩法设计

## 6.1 病人系统

病人随机生成。

每个病人包含：

```ts
interface Patient {
  id: string
  type: 'xray' | 'ct' | 'mri' | 'emergency'
  patience: number
  requiredRoom: string
  processTime: number
  reward: number
  mood: number
}
```

---

## 6.2 病人类型

### 普通胸片（X光）

特点：
- 数量最多
- 处理快
- 分数低

分类：xray 类型，送入 DR拍片室

---

### CT病人

特点：
- 占用时间长
- 分数高
- 需要独占CT室

分类：ct 类型，送入 CT室

---

### MRI核磁病人

特点：
- 占用时间最长
- 分数最高
- 容易堵塞队列

分类：mri 类型，送入 MRI核磁室

---

### 急诊病人

特点：
- 必须优先处理（自动插入队列最前）
- 超时离开扣生命值
- 可送入 DR拍片室 或 CT室

分类：emergency 类型，送入 DR拍片室

# 7.5 小游戏机制（拍片）

每个检查流程包含一个小游戏环节：

## 7.5.1 拍片小游戏

### 游戏类型
滑块对齐游戏（类似拼图/滑块匹配）

### 玩法描述
1. 屏幕上显示一张X光片/CT扫描/MRI图像
2. 图像被切分成几块，并有一定偏移
3. 玩家通过点击方向按钮或拖拽，将图像块滑动到正确位置
4. 对齐完成就算拍片成功

### 视觉设计
```
┌─────────────────────────┐
│     📋 胸片拍摄中       │
├─────────────────────────┤
│  ┌───┬───┬───┐          │
│  │ ▓ │ ▓ │ ▓ │  ←偏移   │
│  ├───┼───┼───┤          │
│  │ ▓ │ ★ │ ▓ │  ★=目标  │
│  ├───┼───┼───┤          │
│  │ ▓ │ ▓ │ ▓ │          │
│  └───┴───┴───┘          │
├─────────────────────────┤
│   [←] [↑] [↓] [→]       │
│      点击移动滑块        │
└─────────────────────────┘
```

### 难度参数
- DR/X光片：3x3网格，宽松判定（偏移<20px算对齐）
- CT片：4x4网格，中等判定（偏移<15px算对齐）
- MRI：4x4网格，中等判定
- 急诊：快速模式，时间限制更短

### 奖励
- 完美对齐（全部偏移<5px）：基础分 × 2
- 良好对齐（平均偏移<15px）：基础分 × 1.5
- 一般对齐：基础分 × 1.0
- 对齐失败：基础分 × 0.5

---

## 7.5.2 报告系统

拍片完成后，系统自动生成一条待写报告，添加到报告清单。

玩家可在报告清单中勾选已完成报告，点击提交获得额外加分。

### 报告提交奖励
- 每提交一份已勾选报告：+5分

---

## 7.5.3 完整检查流程

```
病人到达
    ↓
分配到检查室（等待中...）
    ↓
┌─────────────────┐
│  拍片小游戏      │ ← 滑块对齐
└─────────────────┘
    ↓
检查完成，自动添加待写报告
    ↓
玩家勾选并提交报告
    ↓
获得分数 + 耐心奖励 + 报告加分
病人离开
```

# 8. 核心数值

## 病人耐心值

病人会持续掉耐心。

耐心为0：

- 病人离开
- 扣生命值
- 扣评分

---

## 玩家生命值

默认：

```txt
10 条命
```

病人耐心归零、病人流失：

```txt
-1
```

生命值归零：

```txt
Game Over
```

---

## 分数机制

完成检查获得：

```txt
基础分 + 剩余耐心奖励
```

连续成功：

```txt
Combo 加成
```

# 18. 多端适配设计

## 18.1 适配优先级

本项目必须优先适配：

```txt
手机 > 平板 > PC
```

原因：

- 游戏大概率通过微信、聊天软件、浏览器链接分享
- 目标用户首次体验大概率发生在手机端
- 手机端如果不好玩，PC端完成度再高也没有意义

---

## 18.2 设计原则

### 第一原则：不要强依赖拖拽

PC端可以支持拖拽。

但移动端必须支持：

```txt
点击病人 → 点击设备/房间 → 完成分配
```

避免玩家在手机浏览器里遇到：

- 拖拽误触
- 页面滚动冲突
- 手指遮挡
- 小目标难点

---

### 第二原则：所有可点击区域要足够大

移动端按钮和卡片最小点击区域：

```txt
44px × 44px
```

重要操作按钮建议：

```txt
56px 以上高度
```

---

### 第三原则：单屏完成核心操作

手机端不要让玩家频繁滚动。

核心区域必须在一屏内可完成：

- 查看队列
- 选择病人
- 查看设备状态
- 分配病人
- 查看分数/生命值

---

## 18.3 响应式布局方案

### PC端布局

```txt
顶部：状态栏
左侧：病人队列
中间：影像设备/房间
右侧：提示/任务/事件
底部：操作提示
```

适合宽屏展示完整科室地图。

---

### 平板布局

```txt
顶部：状态栏
上半区：房间/设备
下半区：病人队列
底部：提示信息
```

适合横屏和竖屏。

---

### 手机端布局

```txt
顶部：紧凑状态栏
中部：设备卡片区
下部：病人队列横向滑动
底部：操作提示 / 当前选中病人
```

手机端建议采用：

```txt
卡片式调度界面
```

而不是复杂地图。

---

## 18.4 手机端核心交互

### 推荐交互：点选分配

流程：

```txt
1. 点击病人卡片
2. 病人卡片高亮
3. 可用设备高亮
4. 点击设备卡片
5. 病人进入处理状态
```

---

### 可用设备提示

当选中病人后：

- 可用设备：高亮
- 不可用设备：置灰
- 忙碌设备：显示倒计时

---

### 错误操作反馈

例如：

```txt
“这个病人不能去 CT 室”
“DR 拍片室还在忙”
“急诊快超时了！”
```

手机端反馈必须明显：

- 文案提示
- 轻微震动（可选）
- 卡片抖动动画

---

## 18.5 手机端信息密度控制

手机端每屏最多展示：

```txt
3 个设备卡片
4~5 个病人卡片
1 条提示信息
```

如果病人更多：

```txt
病人队列横向滚动
```

不要把所有信息堆在一个屏幕里。

---

## 18.6 横竖屏策略

### 默认支持竖屏

竖屏是主要体验模式。

### 横屏增强

如果玩家横屏，可以展示更完整的科室地图。

但不能强制玩家横屏。

---

## 18.7 技术实现要求

### CSS 断点建议

```txt
mobile: < 640px
 tablet: 640px ~ 1024px
 desktop: > 1024px
```

---

### Tailwind 响应式示例

```tsx
<div className="grid grid-cols-1 md:grid-cols-2 lg:grid-cols-3">
```

---

### 禁止事项

移动端禁止：

- 依赖 hover
- 过小按钮
- 复杂拖拽作为唯一交互
- 长页面滚动承担核心操作
- 重要信息只在 tooltip 中展示

---

## 18.8 微信/移动浏览器体验

需要注意：

- 首屏加载速度
- 静音策略，音效需要用户点击后再播放
- 避免自动全屏依赖
- 避免过大图片素材
- 使用 WebP/AVIF 压缩素材
- 所有资源走 CDN

---

## 18.9 MVP移动端验收标准

手机端必须满足：

- 首屏 3 秒内可进入游戏
- 单手可以完成核心操作
- 不横屏也能正常玩
- 点选病人和设备不卡顿
- 字体不小于 14px
- 重要按钮不小于 44px
- 病人队列可横滑
- 设备状态一眼可见
